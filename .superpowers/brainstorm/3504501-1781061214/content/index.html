🏗️ 架构方案对比

基于已确认的需求：插件化引擎 · CLI+Skill+Web · MD+Mermaid+HTML · 可配置风险

A: 单体 Python 不推荐

一个 Python 包，CLI 入口调用内部分析模块，报告直接写文件。

开发快，1-2 天可出 MVP
零依赖推理（纯 Python）
插件耦合在核心里，加语言要改核心
Web 服务是后加的外壳，两层维护
大仓库 OOM 风险

B: 分层插件架构 推荐

核心引擎 + 插件接口 + 报告层分离。CLI/Skill/Web 共享同一核心。

插件隔离，新语言 200 行不改核心
核心可独立测试，报告层可替换
CLI/Web/Skill 一套代码
前期多 1-2 天搭框架
插件发现机制需要约定

C: 微服务拆开 过度设计

分析/报告/Web 各一个服务，消息队列通信，可水平扩展。

大仓库可分布式扫描
各组件独立部署升级
MVP 阶段纯浪费，运维成本高
调试链路长，不适合迭代初期
